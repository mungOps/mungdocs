    책의 주요구성 #

    • 패턴명

      • 각 패턴에는 이름이 붙어 있는데, 이 패턴명은 각 장의 제목이기도 하다. 패턴명은 패턴 언어의 핵심이다.

    • 문제

      • 더 넓은 맥락을 제공하고 패턴 영역을 자세하게 설명한다.

    • 해결책

      • 패턴이 쿠버네티스 고유의 방식으로 어떻게 문제를 해결하는지에 관한 내용을 다룬다. 또한, 주어진 패턴의 일부분이거나 관련성 있는 다른 패턴들에 대한 상호 참조도 포함한다.

    • 정리

      • 주어진 맥락에 대한 해결책의 장단점에 대해 정리한다.

    • 참고자료

      • 마지막 절로서, 패턴과 관련된 부가 정보 출처를 제공한다.


    예측 범위 내의 요구사항 #

    • 클라우드 환경에서 애플리케이션 배포, 관리 및 공용을 성공적으로 수행하려면 애플리케이션 자원 요구사항과 런타임 의존성을 명확히 식별하고 정의해야한다.

    • 예측 범위 내의 요구사항 Predictable Demands 패턴이란 하드 런타임 hard runtime 의존성이나 자원 요구사항과는 상관 없이, 애플리케이션 요구사항을 선언하는 방법에 관한 것이다.

    • 컨테이너가 최적의 기능을 수행하는 데 필요한 자원량을 예측하기란 어려워서, 개발자가 테스트를 수행한 후에야 비로소 서비스 구현을 위한 자원 필요량을 알 수 있다.

    • 특정 포트 번호를 사용해야 제대로 작동하는 레거시 서비스 등

    • 애플리케이션 설정 같은 플랫폼 관리 기능이 필요하다.



    예측 필요한 이유 #

    1. 모든 런타임 의존성이 정의되고 자원 요구사항이 계산되면, 쿠버네티스는 가장 효율적인 하드웨어 사용을 위해 클러스터 내에 컨테이너 실행 위치를 지능적으로 결정이 가능

    2. 특별한 서비스 요구사항과 총 서비스 수에 근거해 다양한 환경에 대한 용량 계획을 세우고, 전체 클러스터에 대한 요구사항을 충족하는 비용 효율이 높은 최적의 호스트 프로파일을 만들 수 있다. 성공적인 클러스터 관리를 위해 서비스 자원 프로파일과 용량 계획은 장기적으로 함께 진행



    런타임 의존성 #

    • 가장 일반적인 런타임 의존성은 애플리케이션 상태를 저장하는 데 쓰이는 파일 스토리지
    • 컨테이너 파일 시스템은 일시적이며 컨테이너가 종료되면 삭제
    • 쿠버네티스는 컨테이너 재시작에도 삭제되지 않고 유지되는 다용도의 파드 레벨 스토리지로서 볼륨을 제공


    쿠버네티스 주요 컴포넌트 #

    컴포넌트 역할 설명
    API 서버 (kube-apiserver) 중앙 허브 - 쿠버네티스 클러스터와 통신하는 중앙 허브
    - 클라이언트 요청을 처리하고 인증 및 권한 확인
    - etcd에 상태 저장
    스케줄러 (kube-scheduler) 워크로드 할당 - 새로 생성된 Pod를 클러스터의 적절한 노드에 할당
    - CPU, 메모리 등 리소스 요구 사항 및 정책 고려
    컨트롤러 관리자 (kube-controller-manager) 상태 관리 - 클러스터 상태를 원하는 상태(desired state)로 유지
    - 복제, 노드 상태 감시, Job 관리 등을 담당
    etcd 상태 저장소 - 쿠버네티스 상태 정보를 저장하는 분산 키-값 저장소
    - 모든 상태 변경 사항 기록
    CoreDNS 네트워크 이름 해석 - 클러스터 내 서비스 및 Pod에 대한 DNS 조회 처리
    - 클러스터 서비스 간 통신을 원활하게 만듦
    Kubelet 노드 에이전트 - 각 워커 노드에서 실행되며, API 서버로부터 명령을 받아 Pod를 실행 및 관리
    Kube-Proxy 네트워크 프록시 - 각 노드에서 실행되며, Pod 간 및 외부 트래픽을 라우팅
    - 클러스터 네트워킹 규칙 관리




    관련 데이터플로우 #

    케이스 데이터 요청 데이터 플로우 설명
    HostPath kubectl apply -f로 HostPath 볼륨을 사용하는 Pod 생성 요청 사용자 요청 → API 서버 → etcd → 스케줄러 → 노드 → Kubelet → 로컬 디스크 HostPath 볼륨은 Pod이 노드의 파일 시스템을 직접 마운트하여 사용. 로컬 디스크를 리소스로 활용함.
    NodePort kubectl expose로 NodePort 서비스 생성 요청 외부 요청 → NodePort (노드 포트) → kube-proxy → Pod 클러스터 외부에서 특정 노드 포트를 통해 서비스를 노출. kube-proxy가 트래픽을 적절한 Pod으로 라우팅.
    ConfigMap kubectl create configmap으로 ConfigMap 생성 및 Pod에 마운트 사용자 요청 → API 서버 → etcd → 컨트롤러 → 노드 → Kubelet ConfigMap은 애플리케이션의 환경 변수를 외부에서 설정하기 위한 구성 데이터. Pod에 마운트되어 실행 중에 활용됨.
    Ingress kubectl apply -f로 Ingress 리소스 생성 외부 요청 → Ingress Controller → API 서버 → kube-proxy → Pod Ingress는 HTTP 및 HTTPS 라우팅 규칙을 정의. 외부 트래픽을 URL 기반으로 클러스터 내부의 서비스에 전달.
    PersistentVolumeClaim (PVC) kubectl apply -f로 PVC 생성 요청 사용자 요청 → API 서버 → etcd → 컨트롤러 → 노드 → Kubelet → 스토리지 프로비저너 PVC는 스토리지 요청을 추상화. 프로비저너가 스토리지 클래스에 따라 스토리지를 동적으로 할당하며, Pod에 마운트 가능.
    ClusterIP kubectl expose로 ClusterIP 서비스 생성 요청 클러스터 내부 요청 → ClusterIP → kube-proxy → Pod ClusterIP는 클러스터 내부에서만 접근 가능한 서비스. kube-proxy가 IP 기반으로 트래픽을 라우팅.
    Job kubectl apply -f로 Job 생성 요청 사용자 요청 → API 서버 → etcd → 스케줄러 → 노드 → Kubelet → 컨테이너 실행 완료 후 상태 보고 Job은 특정 작업(예: 데이터 처리)을 한 번 실행한 후 종료되는 워크로드. 스케줄러가 작업을 노드에 배치하고 Kubelet이 실행.
    DaemonSet kubectl apply -f로 DaemonSet 생성 요청 사용자 요청 → API 서버 → etcd → 스케줄러 → 모든 노드 → Kubelet → Pod 실행 DaemonSet은 클러스터의 모든 노드에 동일한 Pod을 실행. 로그 수집, 모니터링 에이전트 등 노드 단위 기능 구현에 활용.
    Deployment kubectl apply -f로 Deployment 생성 요청 사용자 요청 → API 서버 → etcd → 컨트롤러 → 스케줄러 → 노드 → Kubelet → Pod 실행 Deployment는 원하는 상태의 Pod 및 ReplicaSet을 관리. 업데이트 전략을 통해 무중단 배포 지원.


    의존성 설정 #

    • 쿠버네티스의 권장 솔루션은 Confimap
    • 어플리케이션단에서의 설정방법은 환경 변수를 통한 설정과 파일 시스템을 통한 설정 , 2 가지 방법이 았다.


    자원 프로파일 #

    • 쿠버네티스 컨텍스트 내에서의 컴퓨팅 자원 compute resources 이라 함은 컨테이너에 의해 요청되고 , 컨테이너에 할당(스케쥴러)되고 , 컨테이너가 소비할 수 있는 무언가로 정의된다.
    • 이들 자원은 압축 가능 자원 compressible resource ( 예를 들면 CPU 나 네트워크 대역폭처럼 제어 가능한 ) 과 압축 불가능 자원 icompressiale resource ( 예를 들면 메모리처럼 제어 불가능한 ) 으로 분류
    파드 유형 설명 요청(Requests) 제한(Limits) 우선순위 비유
    최선적 (Best-Effort) 요청과 제한이 모두 없는 파드. 자원 보장이 없으며, 노드가 리소스 부족 상태에 빠지면 가장 먼저 종료. 없음 없음 가장 낮음 “내가 필요한 책상을 요구하지 않고, 어디 빈자리가 있으면 앉는 사람” → 자리가 부족하면 가장 먼저 나가야 함.
    확장 가능 (Burstable) 요청과 제한이 설정되어 있지만, 제한이 요청보다 더 큼. 최소한의 자원은 보장되지만, 노드가 자원 부족 상태에 빠지면 최선적 파드가 없을 경우 종료될 가능성이 있음. 있음 있음 (요청보다 큼) 중간 “작은 책상을 요구하지만, 상황이 괜찮으면 큰 책상을 쓰기도 하는 사람” → 최소한 자리는 보장되지만, 자리가 부족하면 나갈 가능성이 있음.
    보장 (Guaranteed) 요청과 제한이 동일하게 설정된 파드. 요청한 리소스를 완전히 보장받으며, 자원이 부족한 상황에서도 가장 마지막에 종료됨. 있음 있음 (요청과 동일) 가장 높음 “내가 필요한 책상을 정확히 요청하고 그 크기를 보장받는 사람” → 언제나 자리가 확보되고, 자리가 부족해도 가장 나중까지 남아 있음.
