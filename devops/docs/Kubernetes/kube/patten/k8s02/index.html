    책의 주요구성 #

    • 패턴명

      • 각 패턴에는 이름이 붙어 있는데, 이 패턴명은 각 장의 제목이기도 하다. 패턴명은 패턴 언어의 핵심이다.

    • 문제

      • 더 넓은 맥락을 제공하고 패턴 영역을 자세하게 설명한다.

    • 해결책

      • 패턴이 쿠버네티스 고유의 방식으로 어떻게 문제를 해결하는지에 관한 내용을 다룬다. 또한, 주어진 패턴의 일부분이거나 관련성 있는 다른 패턴들에 대한 상호 참조도 포함한다.

    • 정리

      • 주어진 맥락에 대한 해결책의 장단점에 대해 정리한다.

    • 참고자료

      • 마지막 절로서, 패턴과 관련된 부가 정보 출처를 제공한다.


    선언적 배포 #

    • 선언적 배포 Declarative Deployment 패턴의 핵심은 쿠버네티스의 디플로이먼트 Deployment 자원
    • 디플로이먼트는 컨테이너 그룹의 업그레이드 및 롤백 프로세스를 캡슐화하며 컨테이너 그룹을 반복적이고 자동화된 동작으로 실행이 가능

    전략별 주요 특징 #

    배포 전략 설명 장점 단점 예시
    롤링 업데이트 새 버전을 점진적으로 배포하며, 기존 버전을 점차적으로 대체. - 다운타임 없이 배포 가능
    - 트래픽 자동 분배    		 - 배포 도중 문제 발생 시 롤백이 어려울 수 있음
    - 모든 사용자에게 변경이 동시에 반영됨    		 기존 v1 애플리케이션에서 v2로 배포하며, 한 번에 1개의 Pod씩 교체. 예: kubectl apply -f deployment.yaml
    고정 배포 기존 버전을 완전히 중단한 후, 새로운 버전을 배포. - 단순한 배포 프로세스
    - 리소스 소모 적음    		 - 배포 시 다운타임 발생
    - 실패 시 복구가 어려움    		 v1 애플리케이션 종료 후, v2 애플리케이션 배포. 예: 기존 Pod 종료 후 새로운 Pod 배포.
    블루-그린 배포 “블루” 환경(현재 버전)과 “그린” 환경(새 버전)을 분리하여 배포 후, 트래픽을 그린 환경으로 전환. - 다운타임 없이 배포 가능
    - 빠른 롤백 가능
    - 테스트 환경과 프로덕션 환경 분리 가능    		 - 두 배포 환경(블루와 그린)을 유지하기 위해 리소스 소모가 큼 v1(블루 환경) 사용 중, v2(그린 환경) 배포 후 트래픽 전환. 문제가 발생하면 블루 환경으로 트래픽 재전환.
    카나리아 릴리스 새 버전을 소수의 사용자에게 배포 후 점진적으로 확장. 문제 발생 시 배포 중단 및 롤백 가능. - 문제를 조기에 발견 가능
    - 점진적 배포로 위험 감소
    - 사용자 그룹별 테스트 가능    		 - 점진적 배포로 시간이 오래 걸릴 수 있음
    - 복잡한 트래픽 관리 필요    		 v1 애플리케이션 사용 중, 5%의 트래픽만 v2로 전환. 안정성이 확인되면 점진적으로 트래픽 증가.
    • 롤링 업데이트: “한꺼번에 다 바꾸지 않고, 하나씩 교체하면서 점검하는 방법.”
    • 고정 배포: “낡은 건물을 부수고 새 건물을 짓는 방법.”
    • 블루-그린 배포: “현재 건물(블루)과 새 건물(그린)을 동시에 유지하다가, 새 건물로 입주자를 옮기는 방법.”
    • 카나리아 릴리스: “새로운 약품을 소수의 사람들에게 먼저 투여해 안전성을 확인한 뒤, 대규모로 투여하는 방법.”
